### 翻译规则（大模型执行时严格遵守）

#### ⚠️ 核心原则：禁止对照表翻译
- 禁止使用任何形式的术语对照表、映射表、词对词替换、硬编码替换。
- 必须完全依赖大模型对上下文的理解，进行自然、流畅的翻译。
- 术语库仅作为背景参考，用于识别"哪些词必须保留原文"，绝不能用于机械替换。

#### 必须原样保留的内容（不翻译）
- 所有代码、命令、文件路径、文件名、环境变量、命令输出
- 所有超链接（URL 及 Markdown 链接格式）
- 所有图片语法（Markdown `![alt](url)` 保持原样，但**后续要将 url 替换为本地路径**）
- Frappe/ERPNext 专有术语（见下方列表）
- 版本号、哈希值、占位符、默认值（`default: 'red'` 中的 `'red'`）

#### 必须保留原文的 Frappe/ERPNext 术语

#### 需要翻译的内容
- 普通英文句子、段落、标题、列表说明文字、表格中的非术语内容。
- 界面提示文本：翻译提示部分，保留默认值原样。

#### 特殊处理
- 图片链接：将原始 URL 替换为本地相对路径 `images/{id}_{文件名}`。注意：这个替换由大模型在翻译时完成，不要留给调度器。

- 保持原文的 Markdown 格式（标题、代码块、列表、表格等）。

            ## 1.Introduction

Frappe provides some basic tooling to quickly write automated tests. There are some basic rules:

Test can be anywhere in your repository but must begin with `test_` and should be a `.py` file.

- Tests must run on a site that starts with `test_`. This is to prevent accidental loss of data.

- Test stubs are automatically generated for new DocTypes.

- Frappe test runner will automatically build test records for dependant DocTypes identified by the `Link` type field (Foreign Key)

- Tests can be executed using `bench run-tests`

- For non-DocType tests, you can write simple unittests and prefix your file names with `test_`.

## Writing Tests for DocTypes

### 2.1. Writing DocType Tests:

- Test cases are in a file named `test_[doctype].py`

- You must create all dependencies in the test file

- Create a Python module structure to create fixtures / dependencies

#### Example (for `test_event.py`):

`# Copyright (c) 2015, Frappe Technologies Pvt. Ltd. and Contributors
# MIT License. See license.txt

import frappe
import frappe.defaults

from frappe.tests.utils import FrappeTestCase

def create_events():
    if frappe.flags.test_events_created:
        return

frappe.set_user(&quot;Administrator&quot;)
doc = frappe.get_doc({
    &quot;doctype&quot;: &quot;Event&quot;,
    &quot;subject&quot;:&quot;_Test Event 1&quot;,
    &quot;starts_on&quot;: &quot;2014-01-01&quot;,
    &quot;event_type&quot;: &quot;Public&quot;
    }).insert()

doc = frappe.get_doc({
    &quot;doctype&quot;: &quot;Event&quot;,
    &quot;subject&quot;:&quot;_Test Event 2&quot;,
    &quot;starts_on&quot;: &quot;2014-01-01&quot;,
    &quot;event_type&quot;: &quot;Private&quot;
    }).insert()

doc = frappe.get_doc({
    &quot;doctype&quot;: &quot;Event&quot;,
    &quot;subject&quot;:&quot;_Test Event 3&quot;,
    &quot;starts_on&quot;: &quot;2014-01-01&quot;,
    &quot;event_type&quot;: &quot;Public&quot;
    &quot;event_individuals&quot;: [{
        &quot;person&quot;: &quot;test1@example.com&quot;
    }]
    }).insert()

frappe.flags.test_events_created = True

class TestEvent(FrappeTestCase):
    def setUp(self):
        create_events()

    def tearDown(self):
        frappe.set_user(&quot;Administrator&quot;)

    def test_allowed_public(self):
        frappe.set_user(&quot;test1@example.com&quot;)
        doc = frappe.get_doc(&quot;Event&quot;, frappe.db.get_value(&quot;Event&quot;,
        {&quot;subject&quot;:&quot;_Test Event 1&quot;}))
        self.assertTrue(frappe.has_permission(&quot;Event&quot;, doc=doc))

    def test_not_allowed_private(self):
        frappe.set_user(&quot;test1@example.com&quot;)
        doc = frappe.get_doc(&quot;Event&quot;, frappe.db.get_value(&quot;Event&quot;,
        {&quot;subject&quot;:&quot;_Test Event 2&quot;}))
        self.assertFalse(frappe.has_permission(&quot;Event&quot;, doc=doc))

    def test_allowed_private_if_in_event_user(self):
        doc = frappe.get_doc(&quot;Event&quot;, frappe.db.get_value(&quot;Event&quot;,
        {&quot;subject&quot;:&quot;_Test Event 3&quot;}))

        frappe.set_user(&quot;test1@example.com&quot;)
        self.assertTrue(frappe.has_permission(&quot;Event&quot;, doc=doc))

    def test_event_list(self):
        frappe.set_user(&quot;test1@example.com&quot;)
        res = frappe.get_list(&quot;Event&quot;, filters=[[&quot;Event&quot;, &quot;subject&quot;, &quot;like&quot;, &quot;_Test Event%&quot;]], fields=[&quot;name&quot;, &quot;subject&quot;])
        self.assertEqual(len(res), 2)
        subjects = [r.subject for r in res]
        self.assertTrue(&quot;_Test Event 1&quot; in subjects)
        self.assertTrue(&quot;_Test Event 3&quot; in subjects)
        self.assertFalse(&quot;_Test Event 2&quot; in subjects)
`
## 2. Running Tests

This function will build all the test dependencies and run your tests.

You should run tests from &quot;frappe_bench&quot; folder. Without options all tests will be run.

bench run-tests

If you need more information about test execution - you can use verbose log level for bench.

bench --verbose run-tests

### Options:

` --app &lt;appname&gt;
 --doctype &lt;doctype&gt;
 --test &lt;specifictest&gt;
 --module &lt;module&gt; (Run a particular module that has tests)
 --profile (Runs a Python profiler on the test)
 --junit-xml-output&lt;pathtoxml&gt; (The command provides test results in the standard XUnit XML format)
`
#### 2.1. Example for app:

All applications are located in folder: &quot;~/frappe-bench/apps&quot;.

#### 2.6 Running Tests without creating fixtures or before_tests hook

- When you are building a feature it is useful to write tests without building test dependencies (i.e build fixtures for linked objects), with `--skip-test-records`

### 3.0 FrappeTestCase

`FrappeTestCase` is Frappe Framework specific TestCase class extended from `unittest.TestCase`. Inherting this class in your tests ensures:

- `frappe.local.flags` and other most used local proxies are reset after test case runs.

- database - a new [database transaction](https://frappeframework.com/docs/v14/user/en/api/database#database-transaction-model) is started before testcase begins and rolled back after tests are finished.

Usage

`# app/doctype/dt/test_dt.py  
  
from frappe.tests.utils import FrappeTestCase  
  
class TestDt(FrappeTestCase):  
 @classmethod  
 def setUpClass(cls):  
 super().setUpClass() # important to call super() methods when extending TestCase.   
 ...
`
## Writing XUnit XML Tests

How to run:
`bench run-tests --junit-xml-output=/reports/junit_test.xml
`
Example of test report:
` &lt;testsuite tests=&quot;3&quot;&gt;
 &lt;testcase classname=&quot;foo1&quot; name=&quot;ASuccessfulTest&quot;&gt;
 &lt;testcase classname=&quot;foo2&quot; name=&quot;AnotherSuccessfulTest&quot;&gt;
 &lt;testcase classname=&quot;foo3&quot; name=&quot;AFailingTest&quot;&gt;
 &lt;failure type=&quot;NotEnoughFoo&quot;&gt; details about failure &lt;/failure&gt;
 &lt;/testcase&gt;
 &lt;/testcase&gt;&lt;/testcase&gt;&lt;/testsuite&gt;
`
It’s designed for the CI Jenkins, but will work for anything else that understands an XUnit-formatted XML representation of test results.

#### Jenkins configuration support:

- You should install xUnit plugin - [https://wiki.jenkins-ci.org/display/JENKINS/xUnit+Plugin](https://wiki.jenkins-ci.org/display/JENKINS/xUnit+Plugin)

- After installation open Jenkins job configuration, click the box named “Publish JUnit test result report” under the &quot;Post-build Actions&quot; and enter path to XML report:

(Example: reports/.xml*)
